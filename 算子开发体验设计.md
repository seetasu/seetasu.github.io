# 算子开发体验设计

## 一句话定位

面向 Ascend C 算子开发者，构建一个以 **Kernel Execution Model（内核执行模型）** 为共同语言的工作台：在设计期解释“为什么应当这样实现”，在读代码时解释“当前实现实际上是什么”，在仿真和上板后解释“硬件实际上发生了什么”。

目标不是把更多底层信息堆给用户，而是缩短这条路径：

```text
现象 / 目标 → 相关信号 → 因果定位 → 方案选择 → 验证与复盘
```

## 用户问题与机会

Ascend C 开发中的核心困难并不只是写 API，而是开发者需要手工在多个断裂的表述之间建立映射：

- 需求里的 shape、dtype、format 与性能目标；
- host / tiling / kernel 的代码实现；
- GM、L1、L0、UB 中的数据驻留和搬运；
- queue、buffer、event、core、tile 的生命周期；
- 编译报错、精度结果、trace、profiling 指标。

今天的工作流通常是先参考样例写出一版，再编译、上板、看 profiler，最后反推 tile、buffer 或同步哪里不对。问题是：代码、设计理由和运行证据彼此不相认。开发者既要读代码，也要手绘流程，还要把运行现象映射回代码。

因此，产品机会不是再做一张“更漂亮的流程图”，而是提供稳定的中间对象，让所有视图、建议与运行数据围绕同一对象工作。

## 核心：Kernel Execution Model

Kernel Execution Model 是一个可追溯、可版本化的执行语义模型，不是单一的大图，也不是 runtime trace 的别名。

它至少包含三层。

| 层级 | 表达的内容 | 典型对象 |
| --- | --- | --- |
| 结构层 | 程序中有哪些可被追踪的资源 | tensor、buffer、queue、event、memory tier、core、tile |
| 逻辑执行层 | 数据和控制在什么依赖下流动 | stage、data edge、dependency edge、loop、slot rotation、lifecycle transition |
| 观测层 | 某一次运行实际测到了什么 | duration、stall、带宽、利用率、峰值驻留、trace event |

其中，**逻辑执行层**是产品的主心智模型：它描述偏序、数据路径和资源状态，而非伪造精确时间。只有在仿真或 profiling 数据存在时，时间、重叠与 stall 才成为模型的观测覆盖层。

### 统一，但不一图到底

同一个模型应按任务投影为不同视图：

- **Pipeline view**：阶段、producer / consumer 依赖和同步。
- **Buffer lifecycle view**：allocate / write / enqueue / wait / consume / release / reuse。
- **Tiling view**：全局 tensor → core → tile → slot 的切分与轮转。
- **Memory path view**：GM、L1、L0、UB 等资源之间的数据移动。
- **Runtime overlay**：对上述对象叠加实际 duration、stall、带宽、峰值占用与正确性信号。

统一的是对象身份和证据链；不统一的是界面。把所有层一次性平铺成“万能大图”会提高复杂性，而不是帮助理解。

### 每条结论必须带证据身份

模型中的对象、边和结论都必须区分来源与可信度：

| 标签 | 含义 | 示例 |
| --- | --- | --- |
| 代码确认 | 可直接定位到源码/API/配置 | `LocalTensor` 位于 `VECCALC`；某次 `DataCopy` 写入它 |
| 静态推断 | 由控制流、事件或规则推导 | Ping/Pong 按 `loopIdx & 1` 交替复用 |
| 估计 / 建议 | 由代价模型或工程规则产生 | double buffer 可能覆盖 CopyIn 等待 |
| 实测 | 来自仿真、benchmark 或 profiling | MTE2 与 Vector 实际重叠比例；UB 峰值 |
| 无法确定 | 当前材料不足 | 真实调度顺序、硬件资源竞争、最终性能根因 |

这是体验可信度的底线：不能因为界面把对象连起来，就把静态依赖误说成真实并行，也不能把模型估计写成 profiling 事实。

## Recommendation Trace：设计期的模型生成器

Recommendation Trace 的职责不是输出一个“最高分方案”，而是让用户审查设计空间如何被约束、规则、偏好与证据逐步收缩。

```text
Operator intent
  → Semantic model
  → Constraint filtering
  → Candidate generation
  → Cost / rule ranking
  → Explainable recommendation
  → Compile / benchmark / profile
  → Evidence feedback
```

它产出的最终推荐应当包含两部分：

1. 一个可执行候选：tile、buffer、pipeline、core split、layout 等选择；
2. 该候选对应的 **目标执行模型**：预期的 buffer 生命周期、数据路径和依赖结构。

推荐不是事实本身。API 合法性、容量限制等是确定性判断；搬运与计算是否值得重叠、三缓冲是否值得等，多数是估计或工程建议；绝对性能只能由实测确认。

### 推荐体验应回答的五个问题

1. 有哪些可控设计变量与候选空间？
2. 哪些候选因 API、对齐、容量、依赖或芯片约束而不可行？
3. 剩余候选在性能、资源余量、复杂度与风险之间如何权衡？
4. 为什么推荐当前方案，为什么不是相邻替代方案？
5. 要用什么编译、正确性与 profiling 信号验证这个选择？

Recommendation Trace 已有概念原型：候选收敛链、Tiling Candidate Atlas、资源过滤、Pipeline Check 与候选代码预览。但当前页面以故事化 mock 数据为主，尚未连接真实源码、编译产物或 runtime 证据。

## Code Recovery：实现期的模型恢复器

Code Recovery 解决的是另一个任务：开发者拿到已有算子时，不必逐行阅读和手绘流程，而是先得到“代码实际表达的执行模型”，再回到源码核验。

它不应被定位成通用 C++ 自动理解，也不能承诺仅靠源码还原真实硬件时序。它的第一职责是恢复：

- buffer / queue / event 的实体与关系；
- 数据搬运、计算、同步和循环形成的逻辑依赖；
- tiling、core 切分、tile loop、slot rotation 与 tail 表达式；
- 每个结论可回跳的代码位置，以及无法恢复的断点。

### 当前受控样例

工作区的 `double buffer学习.asc.cpp` 是 Vector Add 的双缓冲 kernel 片段：两个输入从 GM 搬入 `VECCALC/UB`，执行 `Add`，再写回 GM；通过 Ping/Pong LocalTensor 与 `MTE2 ↔ V`、`V ↔ MTE3` 事件交接实现逻辑上的双缓冲。

它不是 Cube，也不是 Cube–Vector 融合算子：没有 `LoadData`、`Mmad`、L0A/L0B/L0C、Fixpipe 或 AIC→AIV handoff。它很适合作为 Code Recovery 的第一受控输入，因为能清晰验证 buffer、event、loop、slot 与 tail 的恢复能力，但不应被当成覆盖复杂算子能力的证据。

后续样例按复杂度扩展：

| 阶段 | 样例类型 | 需要新增恢复能力 |
| --- | --- | --- |
| 1 | Vector 双缓冲 Add | UB 生命周期、event 配对、线性 tiling、尾块 |
| 2 | Cube MatMul | GM→L1→L0A/B→L0C、LoadData、Mmad、K 累加、二维 core split |
| 3 | Cube + Vector 融合 | AIC/AIV handoff、C→V 路径、跨流水同步、融合资源风险 |

## 二者的关系：共享模型，不急于合并界面

Recommendation Trace 和 Code Recovery 应共享模型契约，但第一阶段不应强行合并为同一个页面或单一任务流。

| 能力 | 主要输入 | 对模型的作用 | 核心用户问题 |
| --- | --- | --- | --- |
| Recommendation Trace | 算子意图、数据契约、芯片约束、用户偏好 | 生成目标执行模型与备选模型 | 应该怎样设计？为什么？ |
| Code Recovery | host / tiling / kernel 源码 | 恢复实现执行模型 | 当前代码究竟实现了什么？ |
| 仿真 / 上板 / Profiling | trace、统计数据、正确性结果 | 向模型写入观测覆盖层 | 实际发生了什么？是否符合预期？ |

未来最有价值的联动不是共享一张流程图，而是基于同一对象做差异：

```text
推荐目标模型  ↔  代码恢复模型  ↔  运行观测模型
         设计偏差          行为 / 性能偏差
```

例如，一个推荐方案的目标是 `CopyIn(n+1)` 与 `Compute(n)` 重叠；Code Recovery 可指出当前 `WaitFlag` 位置是否使这个意图在逻辑上退化；profiling 再验证是否真的存在重叠与等待。这是从“建议—实现—验证”形成闭环，而不是三个孤立工具互相链接。

## 产品形态与阶段

### 阶段 A：理解现有实现

优先做 Code Recovery 静态原型。输入是受控的 Vector 双缓冲样例；输出是逻辑执行模型的 Pipeline、Buffer Lifecycle、Tiling 三种联动视图，并保留源码证据与未解析内容。

成功标准不是“自动解析一切”，而是用户能够在不逐行反推的情况下回答：有哪些 buffer、它们何时被复用、哪些 event 约束了阶段、tile 如何映射到 Ping/Pong 与 tail。

### 阶段 B：设计可审查的候选

将 Recommendation Trace 的候选、约束和解释明确投射为目标执行模型。用户不仅看到 `tile=... / buffer=...`，还可看到该选择预期产生什么 buffer 生命周期和依赖结构。

### 阶段 C：设计—实现差异

把 Code Recovery 输出与一个已选推荐候选对比。产品重点是指出有证据的差异，不自动判定所有差异都是错误：可能是人为优化、版本演进、代码未完成，或推荐假设不适用。

### 阶段 D：运行证据闭环

接入仿真 trace、benchmark、profiling 与正确性结果，为模型对象挂载运行证据。此时才显示真实时间线、实测 overlap、stall、带宽、资源峰值与预测偏差。

## 当前整体进度

| 方向 | 当前状态 | 主要缺口 |
| --- | --- | --- |
| Recommendation Trace 产品叙事与信息架构 | 已形成产品故事、职责拆分与 V3 概念原型 | 仍以 mock 数据和受控案例为主，未接真实模型 |
| Tiling / Pipeline 可视化 | 已有候选 atlas、pipeline、资源与内存路径表达 | 不能从源码自动生成，部分关系是预设数据 |
| Kernel Execution Model | 已完成概念定义与跨阶段定位 | 尚未固化为可交换的数据模型 / schema |
| Code Recovery | 已确定静态优先、规则驱动、Vector 双缓冲样例切入 | 尚未形成可交互原型或解析器 |
| 仿真 / 上板证据回写 | 已明确为观测覆盖层，而不是另一套图 | 尚未对接真实 trace / profiling 数据 |

## 当前设计原则

1. 以问题相关的信号降低复杂性，而不是最大化数据可见性。
2. 设计意图、代码事实、静态推断、模型估计与运行实测必须显式区分。
3. 统一对象模型，不强制统一界面；不同任务需要不同投影。
4. 推荐应提供高质量、可验证的候选集，而非承诺静态阶段的绝对最优。
5. 任何自动恢复都必须显示证据来源、适用条件和未覆盖范围。
6. 成功的体验不是“用户看到了图”，而是用户能更快回答：正在发生什么、为什么发生、下一步看哪里、如何验证。
